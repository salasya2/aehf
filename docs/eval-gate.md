# The eval gate (CI regression check)

aehf can run as a **pull-request gate**: every PR into `main` re-runs the eval
suite, compares the result against a stored baseline, posts a scorecard as a PR
comment, and **fails the check on a statistically significant regression**.

## What "regression" means here

A single case flipping from pass to fail is often just model nondeterminism, not
a real regression. So the gate does not fail on any failing case — it fails only
when the change is **statistically significant** under McNemar's paired test
(p < 0.05) *and* there is at least one newly-failing case. Run-to-run noise does
not trip the check; a real capability drop does.

This is the whole reason the project carries n-sampling, Wilson intervals, and
McNemar's test: so the CI verdict is trustworthy rather than flaky.

## How it runs

The workflow (`.github/workflows/eval.yml`) triggers on `pull_request` into
`main`. It runs on PR open, on every push to an open PR, and on reopen. It does
**not** run on direct pushes to `main`.

Each run:

1. Loads the **baseline** run and the **candidate** run from the results store
   (`.aehf/<sha>/<model>__<judge-version>.json`), keyed by git SHA.
2. Diffs them (`aehf diff BASE_SHA HEAD_SHA`), producing newly-failing /
   newly-passing / unchanged counts plus a McNemar p-value.
3. Writes a markdown scorecard and posts it with `gh pr comment`.
4. Exits non-zero if the diff is a significant regression, failing the check.

## How baselines get into the store

**Runs are generated by a maintainer locally and committed to `.aehf/`.** aehf
does not (yet) record model responses, so a run cannot be reproduced offline in
CI — the agent still calls the live model. Rather than spend API budget and risk
nondeterminism on every PR, the baseline is a committed artifact, the way
golden-file tests commit expected output.

To seed or refresh a baseline:

```bash
git rev-parse HEAD                       # this SHA becomes the baseline key
aehf run examples/calibration_suite.yaml anthropic mock \
    --judgechoice assertion --store .aehf
git add .aehf && git commit -m "eval baseline"
```

`--store` writes one canonical run per `(sha, model, judge-version)`. The command
uses the live model, so `ANTHROPIC_API_KEY` must be set and it costs a few cents.

## Trying it locally

The gate logic is just the `diff` command — you can run it without CI:

```bash
aehf diff <base-sha> <head-sha> --store .aehf --out scorecard.md
echo "exit code: $?"     # 0 = clean, 1 = significant regression, 2 = load error
```

It prints the scorecard to the terminal and (with `--out`) writes it to a file.

## Scope note

The committed-baseline design trades full CI automation for zero cost and
determinism. Fully automated live evaluation in CI would require recording model
responses keyed by conversation state — deliberately out of scope for v0.1 (see
`docs/decision.md`). The gate as built demonstrates the complete mechanism:
store → diff → significance test → scorecard → check status.
